## 2024-05-18 - [Fix Unhandled TypeError in Admin Login Endpoint]
**Vulnerability:** A Denial of Service (DoS) and unhandled exception risk existed in `server/api/admin/login.post.ts`. The endpoint read the `password` property from the JSON body but did not validate its type. A malicious user could send an array or object (e.g., `{"password": []}`) which would pass truthiness checks but cause `crypto.createHash('sha256').update(password)` to throw a synchronous `TypeError [ERR_INVALID_ARG_TYPE]`, resulting in a 500 Internal Server Error instead of a 400 Bad Request.
**Learning:** Even when checking for presence (`if (!password)`) and length (`if (password.length > 1024)`), type confusion can occur if the input is parsed from JSON (where it can be an object or array). Methods like `crypto.update()` strictly require strings or buffers.
**Prevention:** Always validate that the expected scalar type (e.g., `typeof password === 'string'`) is matched before passing user input to strictly-typed Node.js built-ins. Use Zod schemas where possible, or add explicit type guards.

## 2026-03-11 - [Fix SSRF vulnerability in Web Push subscription]
**Vulnerability:** A Server-Side Request Forgery (SSRF) vulnerability existed in the `/api/notifications/subscribe` endpoint. While the web push subscription endpoint was validated with Zod's `z.string().url()`, this only verified the format of the string. An attacker could register an endpoint pointing to `http://localhost`, `169.254.169.254` (cloud metadata), or other internal/private IP ranges. When the server attempted to send a notification, it would make HTTP requests to these restricted locations.
**Learning:** Zod's `.url()` validation is insufficient for endpoints fetched by the server because it does not block loopback, localhost, or RFC1918 private IP ranges. Additionally, synchronous validation using the native Node `URL` parser is preferable to avoid wide-spread refactoring to `.parseAsync()` across all routes that would be required if an asynchronous `.refine()` were used.
**Prevention:** Endpoints fetched by the server must be explicitly validated to enforce HTTPS and strictly block internal/private IP addresses and local hostnames.
